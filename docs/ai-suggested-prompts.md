# AI Suggested Prompts — what "pills" vs "predefined prompt" means, and what was built

This doc explains the feature added to answer the client's request for a
"predefined prompt" on every AI conversation screen, and how it's different
from "pills" (which the client said are already fine).

## 1. The terminology mix-up

Two different things exist (or now exist) in this app, and they were getting
conflated:

| | **Pills** (already existed, client is happy with these) | **Predefined / Suggested Prompts** (new, this doc) |
|---|---|---|
| What it is | Short tap-able reply options the AI offers **mid-conversation** (e.g. "Yes, that's right" / "No, something else", or "Male" / "Female") | Example messages shown **before the user has typed anything**, to help them get started |
| When it appears | Inside an active chat, generated by the AI's state machine as it talks | On the empty/welcome state of a chat screen, before the first message |
| Where it lives in code | `AiMessage` rows with `type = 'pills'`, generated by `App\Services\WorkspaceConversationService` | New `ai_suggested_prompts` table, fetched once via a dedicated endpoint |
| Example | `["Yes, that's right", "No, something else"]` | `"Find posts about hiking trips"`, `"I want to sell my old bicycle"` |

Pills are dynamic and conversation-specific — they can't be "predefined" because
they depend on what stage the conversation is in. What the client actually
wants is a **static list of example starter messages**, shown as tappable
chips before the user types their first message, so they know what kind of
thing they can ask for. That's what got built.

## 2. Where this applies — the AI chat APIs in this app

| API | Purpose | Suggested-prompt `context` |
|---|---|---|
| `POST /api/feed/ai-chat` | Conversational feed search (greets, asks to clarify, or returns results) | `feed_search` |
| `POST /api/conversations` + `POST /api/conversations/{slug}/messages` | AI-guided post/ad/event creation chat | `workspace_conversation` |

`feed_search` and `workspace_conversation` share one endpoint (`/api/ai/suggested-prompts`),
distinguished by a `context` query param, so the frontend calls the same API
for every AI chat screen it has.

> `POST /api/feed/ai-search` (one-shot keyword search) was removed — it
> predated `/feed/ai-chat` and did the same job without the conversational
> framing. Use `/feed/ai-chat` for all feed search; the `results` type on its
> response carries the matching posts.

## 3. What was built

- **Migration** `database/migrations/2026_07_19_090000_create_ai_suggested_prompts_table.php`
  → table `ai_suggested_prompts`: `context`, `workspace_id` (nullable), `label`,
  `prompt`, `sort_order`, `is_active`.
- **Model** `app/Models/AiSuggestedPrompt.php` — `active()` / `forContext()` scopes.
- **Seeder** `database/seeders/AiSuggestedPromptSeeder.php` — starter content
  (5 prompts for `feed_search`, 5 for `workspace_conversation`, some scoped to
  a specific workspace e.g. "I want to sell my old bicycle" → Market Place
  only). Registered in `DatabaseSeeder`.
- **API** `GET /api/ai/suggested-prompts` — `app/Http/Controllers/API/AiSuggestedPromptController.php`.

### Endpoint

```
GET /api/ai/suggested-prompts
GET /api/ai/suggested-prompts?context=feed_search
GET /api/ai/suggested-prompts?context=workspace_conversation
GET /api/ai/suggested-prompts?context=workspace_conversation&workspace=market_place
```

Auth: same `auth:api` (JWT) middleware as every other endpoint in this group.

**No `context` param** → everything, grouped by context:
```json
{
  "status": true,
  "message": "Suggested prompts fetched successfully",
  "data": {
    "feed_search": [
      { "id": 1, "label": "Hiking trips", "prompt": "Find posts about hiking trips", "workspace_slug": null }
    ],
    "workspace_conversation": [
      { "id": 6, "label": "Share a post", "prompt": "I want to share photos from my trip", "workspace_slug": null },
      { "id": 8, "label": "Sell something", "prompt": "I want to sell my old bicycle", "workspace_slug": "market_place" }
    ]
  },
  "code": 200
}
```

**With `context`** → a flat array (what the frontend will use in practice on
any single chat screen):
```json
{
  "status": true,
  "message": "Suggested prompts fetched successfully",
  "data": [
    { "id": 1, "label": "Hiking trips", "prompt": "Find posts about hiking trips", "workspace_slug": null },
    { "id": 2, "label": "Weekend events", "prompt": "Show me events happening this weekend", "workspace_slug": null }
  ],
  "code": 200
}
```

Add `workspace=<slug>` (e.g. `market_place`, see `Workspace::SLUG_*` constants)
for `workspace_conversation` to also include prompts scoped to that specific
workspace (workspace-agnostic prompts always come back regardless).

## 4. How the frontend should use it

1. On opening any AI chat screen, call `GET /api/ai/suggested-prompts?context=<feed_search|workspace_conversation>` and render each item as a tappable chip using `label`.
2. When the user taps a chip, send its `prompt` value as the normal message field to the existing chat endpoint — exactly as if the user had typed it:
   - Feed search chat → `POST /feed/ai-chat` body `{ "message": "<prompt>" }`
   - Conversation → `POST /conversations/{slug}/messages` body `{ "message": "<prompt>" }`
3. No new "pill" handling is needed on the frontend for this — it's just pre-filling/sending the existing message field. Pills (mid-conversation options) keep working exactly as before, untouched.

## 5. Adding or editing prompts later

There's no admin UI for this yet — content is seed-managed for now:

- Edit `database/seeders/AiSuggestedPromptSeeder.php` and re-run:
  `php artisan db:seed --class=AiSuggestedPromptSeeder` (it's idempotent — `updateOrCreate` keyed on `context` + `prompt`, safe to re-run).
- Or edit rows directly in the `ai_suggested_prompts` table (`label`, `prompt`, `sort_order`, `is_active`, `workspace_id`).

If the client ends up wanting to edit these themselves without a developer,
the natural next step is a small admin CRUD page mirroring the existing
**Workspaces** admin screen (`app/Http/Controllers/Web/Backend/WorkspaceController.php`
+ `resources/views/backend/layouts/workspaces/*`) — not built yet, flagging
it here in case it's wanted.
